Complete P-CNN FPGA Vitis HLS

CNN Digit Detector

Hardware ML Project PA-StarLite · Artix-7 XC7A200T Vitis HLS · Vivado · PyTorch

A complete end-to-end pipeline that trains a 2D Convolutional Neural Network in PyTorch, exports its weights as fixed-point C++ constants, synthesises the network logic into hardware via Vitis HLS, and deploys the resulting bitstream to an Artix-7 FPGA — with live digit recognition displayed over HDMI.

Target DevicePA-StarLite (Artix-7 XC7A200T · 215 K LUTs · 740 DSPs · 13 Mb BRAM)
ToolchainVitis HLS · AMD Vivado · PyTorch · Python (sklearn)

Overview

The design bridges the software-to-hardware boundary in four stages: training a compact CNN model in software, exporting baked-in weights, synthesising the forward pass into RTL, and integrating the result into a Vivado block design with a custom Verilog top-level. The digit ROM stores 1,797 pre-scaled test images directly in on-chip BRAM — navigated with physical push-buttons — and the CNN result is rendered as a bar chart on the HDMI monitor in real time.

CNN Hardware Pipeline

The forward pass is a three-layer network: a 3×3 convolution with ReLU, a 2×2 max-pooling, and a fully-connected linear classifier for 10 digit classes. All intermediate buffers are dual-port BRAMs partitioned across channel dimensions for II=1 pipelining.

Layer 1 — 2D Convolution & ReLU

A 3×3 sliding window (stride=1, padding=1) is applied to the single-channel 8×8 input, producing four feature maps. Weights are baked into ROM constants (conv_w[4][1][3][3]) exported from PyTorch. Each output pixel is biased (conv_b[4]) and clipped to zero by the ReLU activation, then written to a dual-port BRAM partitioned along the channel dimension — enabling II=1 with the #pragma HLS UNROLL on the outer channel loop.

Layer 2 — 2×2 Max Pooling

Non-overlapping 2×2 windows reduce each of the four 8×8 feature maps to 4×4, yielding a 4×4×4 pooled tensor stored in a second BRAM. The hardware reads two adjacent rows simultaneously using both read ports of the dual-port BRAM and resolves the 4-pixel maximum through a hierarchical comparator chain, keeping the pool pipeline at II=1.

Layer 3 — Fully Connected Classifier

The 64-element flattened vector (from 4×4×4) is multiplied against 64 weight-ROM parameters per output class. The top-level module pre-loads all 64 values into parallel registers before launching the FC pipeline, which runs 10 dot products in a single pipeline iteration — each dot product accumulating 64 products and adding one bias ROM value, writing to the prediction[10] output interface.

Precision & Fixed-Point Arithmetic

All data paths use ap_fixed<16,6> — 16-bit words with 6 integer bits (including sign, covering −32 to +31.999) and 10 fractional bits (resolution ≈ 0.001). This format was validated by co-simulating the HLS bit-accurate model against the floating-point PyTorch outputs.

High-Level Synthesis Design

The CNN forward pass is described in cnn.cpp using Vitis HLS pragmas for maximum throughput. Key directives:

// Convolution outer-loop fully unrolled — 4 channels computed in parallel
conv_label_oc: for (int oc = 0; oc < 4; oc++) {
    #pragma HLS UNROLL
    conv_label_r: for (int r = 0; r < 8; r++) {
        conv_label_c: for (int c = 0; c < 8; c++) {
            #pragma HLS PIPELINE II=1
            // 3x3 window accumulation with zero-padding
        }
    }
}

// Intermediate buffers partitioned to allow concurrent read/write
data_t conv_out[4][8][8];
#pragma HLS ARRAY_PARTITION variable=conv_out complete dim=1

data_t pool_out[4][4][4];
#pragma HLS ARRAY_PARTITION variable=pool_out complete dim=1

The channel dimension is fully partitioned in both BRAM buffers so all four channels can be read or written in a single clock cycle, enabling the convolution and pooling loops to sustain II=1 without memory conflicts. The HLS project is built using the run_hls.tcl script and the complete Vivado integration is automated via build_cnn.tcl.

RTL Architecture

The HLS synthesis generates a module hierarchy under cnn.v orchestrated by a 20-state Control FSM. The FSM schedules overlapping convolution and pooling pipelines, then performs a parallel register-loading phase before firing the fully-connected classifier.

Module Hierarchy

  • cnn.v (Top-Level) — Control FSM, memory bus routing multiplexers, and AP handshake (ap_start / ap_done).
  • 8× conv_out BRAM — Dual-port AUTO-inferred BRAMs (cnn_conv_out_RAM_AUTO_1R1W.v) holding the 4×8×8 activated feature maps across two sub-banks per channel.
  • 4× pool_out BRAM — Dual-port BRAMs (cnn_pool_out_RAM_AUTO_1R1W.v) for the 4×4×4 pooled maps.
  • 4× Convolution Pipeline Modules — Each pipeline module drives a row/column loop controller and a DSP multiply-accumulator, reading pixel data through dual-port image interfaces and writing to a pair of conv_out BRAMs.
  • 4× Max Pooling Pipeline Modules — Each reads from a pair of conv_out BRAMs using all four read ports simultaneously, resolves the 2×2 maximum, and writes to one pool_out BRAM.
  • 1× Fully Connected Pipeline — Dot-product engine backed by 64 weight-parameter ROMs (fc_w_0_ROM … fc_w_63_ROM), 1 bias ROM, and a parallel DSP MAC array (cnn_mac_muladd_...).

FSM Schedule

The 20-state FSM implements an overlapping execution schedule to maximise silicon utilisation. Convolution for one filter pair is launched while pooling from the previous pair runs in parallel:

  • States 1: Idle — wait for ap_start
  • States 2–10: Pipelined CONV+POOL — 4 convolution passes interleaved with 4 pooling passes
  • States 11–19: Parallel Register Loading — 8 values fetched per cycle to fill 64 FC input registers
  • State 20: FC Classification pipeline — assert ap_done when complete

Memory Bus Multiplexing

Because both the convolution and pooling pipelines share the same conv_out BRAMs (one writes, one reads), the top-level instantiates address, write-enable, and chip-enable multiplexers controlled by the active FSM state. This prevents bus contention while allowing the two pipelines to overlap in alternating states.

Hardware Demonstration

The synthesised bitstream was programmed to the PA-StarLite development board (Artix-7 XC7A200T) using a custom program_cnn.tcl script. The digit dataset is stored in on-chip BRAM via digits_rom.v (1,797 pre-loaded images). Two push-buttons cycle through the dataset, a UART port streams the predicted class to a terminal, and the HDMI output renders the 8×8 input pixel grid alongside a bar chart of the 10 class logits.

Implementation Summary

  • Clock — 200 MHz input differential clock (DIFF_SSTL135), constrained at 5.0 ns period. Timing closure achieved with no negative slack.
  • Arithmetic precision — ap_fixed<16,6> throughout. Bit-accurate co-simulation matches PyTorch floating-point reference outputs.
  • Control FSM — 20-state pipelined schedule with overlapping CONV/POOL phases and parallel register pre-loading for the FC stage.
  • Memory — All intermediate feature maps stored in Vivado-inferred dual-port BRAMs; weight and bias constants synthesised as dedicated ROM resources from weights.h.
  • I/O — HDMI TMDS output via 3 differential data lanes + 1 differential clock (Bank 14); UART TX at 3.3 V LVCMOS for serial debug; 2 push-buttons for dataset navigation.
